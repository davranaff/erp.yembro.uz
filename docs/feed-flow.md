# Модуль «Корма» — обзор флоу

**Статус:** актуально на 2026-04-30
**Аудитория:** новый разработчик / технолог / бухгалтер, желающие понять как устроен модуль

---

## Краткое описание в одну строку

Покупаем сырьё → лежит на складе (там сохнет) → мешаем по рецепту → получаем готовый корм (тоже сохнет) → проверяем лабораторно → отдаём птице.

```
                  ПОСТАВЩИК
                      ↓
           ┌──── Приёмка сырья ────┐
           │  [Дюваль-поправка]    │
           │  Карантин → Available │
           └───────────┬───────────┘
                       │
                       ↓
           ┌─── Хранение на складе ───┐

           └───────────┬──────────────┘
                       │
                       ↓
           ┌──── Замес ────┐
           │  Рецепт vN    │
           │  FIFO подбор  │

                   │
                   ↓
        ┌── Партия готового корма ──┐

                     ↓
              [Маточник / Откорм]
              кормление птицы
```

## 1. Рецепт — «по какой формуле мешать»

Технолог заводит **рецептуры** — это рецепты комбикорма.

Пример: рецепт «Стартёр для бройлера»
- Кукуруза — 50%
- Шрот соевый — 25%
- Пшеница — 15%
- Премикс — 10%

У рецепта могут быть **версии** (v1, v2, v3) — если технолог корректирует пропорции, старые партии всё равно знают по какой формуле их сделали.

**Где:** UI `/feed` → таб «Рецепты»
**Backend:** `apps.feed.models.Recipe`, `RecipeVersion`, `RecipeComponent`
**API:** `GET/POST/PATCH /api/feed/recipes/`, `recipe-versions/`, `recipe-components/`

---

## 2. Приёмка сырья — «приехала фура»

Поставщик привёз пшеницу. Заведующий складом:

1. Заводит партию: 10 000 кг пшеницы, влажность 18%, цена 2000 сум/кг
2. Система применяет **поправку по Дювалю** (если влажность выше базы) — записывает зачётный вес 9 535 кг
3. Партия попадает в **карантин** (защита от непроверенного сырья)
4. Лаборант проверяет качество → жмёт «Снять карантин» → статус становится `available`

С этого момента партия:
- ✅ может быть использована в замесе
- ✅ автоматически попадает под профиль усушки (создаётся при первой партии этой номенклатуры)

**Где:** UI `/feed` → таб «Сырьё»
**Backend:** `apps.feed.models.RawMaterialBatch`, статусы `quarantine` → `available` → `rejected` / `depleted`
**API:** `POST /api/feed/raw-batches/`, actions `release_quarantine` / `reject_quarantine`
**Дюваль:** `apps.feed.services.shrinkage.compute_settlement()` — чистая функция без БД

## 3. Хранение — «партия лежит на складе»

Между приёмкой и расходом партия может пролежать несколько недель. За это время:

### Усушка при хранении (Этап 2 — реализовано)

Каждую ночь в **02:00 (Asia/Tashkent)** Celery-воркер прогоняет алгоритм:

1. Берёт все активные партии (`available`, `current_quantity > 0`)
2. Для каждой ищет подходящий **профиль усушки** (`FeedShrinkageProfile`):
   - Точная пара `(nomenclature, warehouse)` → побеждает
   - Иначе — общий профиль для номенклатуры (`warehouse=NULL`)
3. Считает компаундно: `loss = current × pct × N_periods`
4. Обновляет `FeedLotShrinkageState` (накопленные потери)
5. Создаёт `StockMovement(kind=shrinkage)` — отдельный тип движения, видимый во всех отчётах склада
6. Уменьшает `RawMaterialBatch.current_quantity`

**Где:** UI `/feed/shrinkage-profiles` (CRUD профилей + drawer'ы), `/reports/feed-shrinkage` (отчёт)
**Backend:** `apps.feed.services.shrinkage_runner` — алгоритм, `apps.feed.tasks.apply_feed_shrinkage_task` — Celery beat
**API:** `/api/feed/shrinkage-profiles/`, `/api/feed/shrinkage-state/` (+ actions `apply` / `reset` / `history`)
**Подробнее:** `docs/feed-shrinkage-spec.md`

## 4. Замес — «делаем корм из сырья»

Технолог планирует производство:

1. Создаёт **задание на замес** (`ProductionTask`):
   - Рецепт: «Стартёр v2»
   - План: 1000 кг
   - Когда: завтра 08:00
   - Линия замеса: «Линия №1»

2. Система **автоматически подбирает партии сырья** по FIFO (старые → первыми) для каждого ингредиента рецепта. Это компоненты задания (`ProductionTaskComponent`).

3. В назначенный день оператор жмёт «Провести замес»:
   - Указывает фактический выпуск (например 985 кг — на 15 кг меньше плана из-за потерь)
   - Указывает склад готовой продукции и бункер хранения

4. Что происходит автоматически:
   - 📦 Сырьё **списывается** со склада (`StockMovement(kind=outgoing)`)
   - 🍲 Создаётся **партия готового корма** (`FeedBatch`) на 985 кг
   - 💰 Считается **себестоимость**: (стоимость списанного сырья) ÷ 985 = X сум/кг
   - 📒 Делается **проводка в бухгалтерии** (`JournalEntry`: Дт «Готовая продукция» / Кт «Сырьё»)
   - ✅ Задание переходит в статус `done`

**Где:** UI `/feed` → таб «Задания» → «+ Новое задание» → «Провести»
**Backend:** `apps.feed.models.ProductionTask`, `ProductionTaskComponent`, `FeedBatch`
**Сервисы:** `services/copy_components.py` (FIFO подбор), `services/execute_task.py` (проведение)
**API:** `POST /api/feed/production-tasks/`, action `execute`

## 5. Лабораторный контроль — «можно ли скармливать»

Готовая партия корма попадает в статус `quality_check` (на лабораторном контроле). Лаборант:

- Берёт пробу
- Проверяет белок, жир, влажность, токсины
- Жмёт **«Одобрить паспорт»** → партия `approved` → готова к расходу
- Или **«Отклонить»** → `rejected` (с причиной)

Параллельно: `LabResult` — отдельная сущность с детальными показателями (можно привязать к партии сырья или готового корма через GenericForeignKey).

**Где:** UI: drawer `FeedBatch` → кнопки «Одобрить паспорт» / «Отклонить паспорт»
**Backend:** `apps.feed.models.FeedBatch.quality_passport_status`, `LabResult`
**API:** action `approve_passport` / `reject_passport` на `/api/feed/feed-batches/{id}/`

## 6. Использование — «корм идёт на птицу»

Партия `approved` корма доступна для расхода в другие модули:
- **Маточник** (родительское стадо)
- **Фабрика откорма** (бройлеры)
- **Продажи** (если корм отгружают на сторону)

Когда зоотехник в фабрике откорма списывает корм на партию бройлеров — `FeedBatch.current_quantity_kg` уменьшается. Себестоимость корма автоматически попадает в стоимость партии цыплят (через цепочку `FeedConsumption` → `FeedlotBatch.cost`).

**Где:** UI других модулей (`/feedlot`, `/matochnik`, `/sales`)
**Backend:** различные сервисы расходования + `StockMovement(kind=outgoing)` с reference на партию готового корма

## Два разных понятия «усушка» — не путать

| Аспект | Дюваль (приёмка) | Профили (хранение) |
|---|---|---|
| Что описывает | Влажность *внутри* партии при приёме | Скорость усушки в условиях склада |
| Когда применяется | Один раз при приходе | Каждую ночь, постоянно |
| Где задаётся | Лаборант вводит влажность в форме приёмки | Технолог настраивает профиль |
| Результат | Зачётный вес = брутто × (1 − %) | Постепенное уменьшение `current_quantity` |
| Поле / модель | `RawMaterialBatch.shrinkage_pct` (одно число) | `FeedShrinkageProfile` + `FeedLotShrinkageState` |
| Движение | Не создаёт `StockMovement` (просто корректирует начальную массу) | Создаёт `StockMovement(kind=shrinkage)` |
