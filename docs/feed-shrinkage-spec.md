Пользователь не настраивает усушку вручную — она работает из коробки. Если профиль был соз­дан и потом soft-deleted — повторно не пересоздаётся (уважаем выбор). ## 1. Зачем это нужно

Сырьё (пшеница, кукуруза, шрот) и готовый корм физически теряют вес на складе: испарение влаги, усушка, пыль. Сейчас в системе остатки считаются как «сколько пришло минус сколько израсходовано» — и через месяц отчёт по складу расходится с реальностью на 2–5% (иногда больше). Бухгалтеру приходится делать ручные корректировки через инвентаризацию, а руководитель не видит реальную себестоимость корма, потому что усушка не разносится на продукцию.

Нужно дать пользователю инструмент: *«для этого ингредиента каждые 7 дней списывай 0.8% веса, но не больше 5% от первоначальной партии»* — и система сама будет вести учёт.

## 2. Какую задачу решаем

1. Пользователь заводит **профиль усушки** на ингредиент или тип готового корма: процент, период, верхний предел.
2. Каждую ночь фоновый воркер проходит по активным партиям и списывает положенный процент.
3. Списание попадает в **журнал движений** (`stock_movements` с новым `movement_kind = 'shrinkage'`), поэтому любой отчёт по остаткам видит его автоматически.
4. Отдельная страница показывает: «сколько веса потеряно за период», «какая партия какой процент уже потеряла», «остались ли партии на верхнем пределе, чтобы прекратить списывать».
5. Для расчёта себестоимости готового корма — усушка сырья до момента его использования прибавляется к стоимости партии, усушка готового корма после производства — к общей списанной массе.

## 3. Ключевые понятия (глоссарий)

| Термин | Что значит |
|---|---|
| **Профиль усушки** | Правило: «N% каждые M дней, максимум K%, остановиться через L дней». Привязан либо к `feed_ingredient_id` (сырьё), либо к `feed_type_id` (готовая продукция). |
| **Партия (lot)** | Конкретное поступление — `feed_raw_arrivals` строка для сырья или `feed_production_batches` строка для готового корма. У каждой партии своя начальная дата отсчёта. |
| **Остаток партии** | Сколько веса ещё не списано (кг). Вычисляется как `начальный_вес − сумма_расходов − сумма_усушки`. |
| **Накопленная усушка** | Сколько уже списано по профилю с момента поступления. Контроль, чтобы не превысить `max_percent`. |
| **Цикл списания** | Одно срабатывание воркера: для каждой партии считает, сколько полных периодов прошло с последнего списания, и добавляет соответствующую долю. |

## 4. Модель данных

### 4.1 Таблица `feed_shrinkage_profiles`

Конфигурация «как усыхает». Один профиль на ингредиент/тип + опциональная привязка к складу (на некоторых складах суше/влажнее).

| Поле | Тип | Описание |
|---|---|---|
| `id` | UUID PK | |
| `organization_id` | UUID FK → organizations | |
| `target_type` | enum('ingredient', 'feed_type') | К чему применяется |
| `ingredient_id` | UUID FK → feed_ingredients, nullable | Если target_type='ingredient' |
| `feed_type_id` | UUID FK → feed_types, nullable | Если target_type='feed_type' |
| `warehouse_id` | UUID FK → warehouses, nullable | NULL = «для всех складов» |
| `period_days` | INT | Каждые сколько дней применяется процент (напр. 7) |
| `percent_per_period` | NUMERIC(6,3) | Сколько % от **текущего** остатка списывается за один период |
| `max_total_percent` | NUMERIC(6,3) nullable | Верхний предел накопленной усушки на партию; NULL = без предела |
| `stop_after_days` | INT nullable | Не списывать дольше N дней от поступления; NULL = без ограничения |
| `starts_after_days` | INT default 0 | Первые N дней не усыхает (грейс-период) |
| `is_active` | BOOL default true | |
| `note` | TEXT nullable | |
| `created_at / updated_at` | TIMESTAMPTZ | |

**Ограничения:**
- Ровно одно из `ingredient_id` / `feed_type_id` заполнено (CHECK).
- `period_days > 0`, `percent_per_period >= 0 AND percent_per_period <= 100`.
- Уникальность `(organization_id, target_type, ingredient_id, feed_type_id, warehouse_id)` с учётом NULL как «все склады» — решается уникальным частичным индексом.

### 4.2 Таблица `feed_lot_shrinkage_state`

Состояние усушки по конкретной партии. Создаётся при первом срабатывании воркера для партии, обновляется каждое списание.

| Поле | Тип | Описание |
|---|---|---|
| `id` | UUID PK | |
| `organization_id` | UUID FK | |
| `lot_type` | enum('raw_arrival', 'production_batch') | Какая таблица в `lot_id` |
| `lot_id` | UUID | `feed_raw_arrivals.id` или `feed_production_batches.id` |
| `profile_id` | UUID FK → feed_shrinkage_profiles | Какой профиль применяем |
| `initial_quantity` | NUMERIC(16,3) | Сколько кг было при поступлении/производстве |
| `accumulated_loss` | NUMERIC(16,3) default 0 | Сколько кг уже списано по усушке |
| `last_applied_on` | DATE | Дата последнего цикла списания |
| `is_frozen` | BOOL default false | Достигнут `max_total_percent` / `stop_after_days` → больше не трогаем |

**Ограничения:**
- Уникальность `(lot_type, lot_id)` — одна партия = одно состояние.
- `accumulated_loss <= initial_quantity`.

### 4.3 Новый `movement_kind` в `stock_movements`

В CHECK `ck_stock_movement_kind_allowed` добавляется значение `'shrinkage'`. Само движение создаётся воркером за каждый цикл списания — это даёт полную прослеживаемость и автоматически «вычитается» из остатков во всех существующих отчётах.

Поля движения при усушке:
- `movement_kind = 'shrinkage'`
- `reference_table = 'feed_lot_shrinkage_state'`
- `reference_id = state.id`
- `note = "Усушка {percent}% за период {from}–{to}"`

### 4.4 Миграция и сиды

Миграция Alembic:
1. Создаёт обе таблицы с индексами.
2. Добавляет значение `'shrinkage'` в CHECK `stock_movements.movement_kind`.
3. Добавляет permissions: `feed_shrinkage_profile.{read, create, write, delete}`, `feed_shrinkage_run.{read, execute}`.
4. Привязывает их к ролям `admin` и `manager_feed`.

## 5. Логика списания (резолвер)

### 5.1 Алгоритм воркера

Раз в сутки (cron `0 2 * * *`, 2:00 ночи Ташкент → 21:00 UTC) задача `apply_feed_shrinkage_task` выполняет следующее для каждой организации:

```
для каждой активной партии сырья / готового корма:
  найти профиль (точная пара ingredient+warehouse, иначе ingredient с warehouse=NULL)
  если профиля нет → пропустить
  если state.is_frozen → пропустить
  days_since_start = today - lot.arrived_on  (или production_batch.finished_on)
  если days_since_start < profile.starts_after_days → пропустить
  если profile.stop_after_days и days_since_start > stop_after_days → заморозить state
  days_since_last_apply = today - (state.last_applied_on или lot.arrived_on)
  полных_периодов = days_since_last_apply // profile.period_days
  если полных_периодов == 0 → пропустить

  текущий_остаток = initial_quantity - accumulated_loss - сумма_расходов_по_партии
  если текущий_остаток <= 0 → заморозить state

  # применяем процент компаундно за каждый период
  потеря = 0
  остаток = текущий_остаток
  для i в range(полных_периодов):
    дельта = остаток * profile.percent_per_period / 100
    если profile.max_total_percent:
      оставшийся_лимит = initial_quantity * max_total_percent / 100 - accumulated_loss - потеря
      если дельта > оставшийся_лимит:
        дельта = оставшийся_лимит
        заморозить_после = true
    потеря += дельта
    остаток -= дельта

  создать stock_movement(kind='shrinkage', quantity=потеря, ...)
  state.accumulated_loss += потеря
  state.last_applied_on = today
  если заморозить_после → state.is_frozen = true
```

### 5.2 Компаунд vs линейное списание

Выбрал **компаундное** (от текущего остатка), не линейное (от начального). Причина: физически усушка пропорциональна тому, сколько воды ещё в продукте; после первого «слива» остаток сухее и теряет меньше. Это даёт асимптотическое поведение, что ближе к реальности.

Линейное (фиксированный % от начальной массы) можно включить опцией `calc_mode = 'linear' | 'compound'` в профиле — отложим до v2, если пользователям будет проще мыслить «каждый день −0.1 кг».

### 5.3 Ручной пересчёт

Эндпойнт `POST /feed/shrinkage/apply` с параметрами `{ on_date?: date, lot_id?: UUID }` — позволяет админу прогнать алгоритм на конкретную дату (для восстановления при простое воркера) или для одной партии (после изменения профиля). Идемпотентность: повторный запуск на ту же дату ничего не добавит, потому что `last_applied_on` уже эта дата.

## 6. API endpoints

Под префиксом `/feed/shrinkage-profiles`:

- `GET /` — список с фильтрами `target_type`, `ingredient_id`, `feed_type_id`, `is_active`.
- `POST /` — создать профиль.
- `GET /{id}` — прочитать.
- `PATCH /{id}` — обновить (`is_active`, проценты, периоды).
- `DELETE /{id}` — мягкое удаление через `is_active=false` (жёсткое запрещаем, потому что ссылается `feed_lot_shrinkage_state`).

Под префиксом `/feed/shrinkage-state`:

- `GET /` — состояние по партиям, фильтры: `lot_type`, `lot_id`, `warehouse_id`, `is_frozen`.
- `GET /{id}` — детально: профиль, история движений, текущий остаток, накопленная усушка в кг и %.
- `POST /apply` — ручной триггер расчёта (см. 5.3), `admin`-only.
- `GET /report` — сводный отчёт: за период `date_from/date_to` группировка по ингредиенту или по складу, сумма списанной массы и её стоимость (берётся из `unit_price` прихода).

Все новые пермишены: `feed_shrinkage_profile.{read, create, write, delete}` + `feed_shrinkage_run.{read, execute}`.

## 7. Влияние на существующие модули

**`feed_raw_arrivals`** — без изменений, но при создании прихода воркер автоматически подхватит партию в следующий цикл, если под ингредиент есть активный профиль.

**`feed_production_batches`** — добавляется реляция `shrinkage_state` (back_populates). Отсчёт усушки ведётся от `finished_on`, а не `started_on` (производство уже может занять несколько дней).

**`feed_raw_consumption`, `feed_consumption`, `feed_product_shipments`** — проверка «хватает ли остатка» должна учитывать уже списанную усушку. Это происходит автоматически, потому что все расчёты идут через агрегирование `stock_movements`, куда усушка попадает как `outgoing`-подобная запись.

**Себестоимость готового корма (feed production)** — при закрытии партии `feed_production_batches` фактическая стоимость = сумма(`feed_raw_consumption.quantity × unit_price_партии`). Если сырьё усохло до потребления, `quantity` уже учитывает реальный израсходованный объём, но стоимость партии сырья остаётся исходной. Поэтому эффект усушки сырья = (усохший_кг × unit_price) становится отдельной строкой «Потери от усушки» в P&L, не попадающей в себестоимость готового продукта.

**Себестоимость готового корма после производства** — усушка готового корма после `finished_on` списывается на «Потери по готовой продукции», корректируя валовую маржу отгрузок (если отгружено меньше, чем произведено минус расход — разница это усушка + брак).

**Финансы (`cash_transactions`, ранее добавленный резолвер курса валют)** — никак не затрагивается. Усушка — это списание натурального веса, без движения денег.

## 8. Frontend

### 8.1 Новая страница «Настройки усушки» (`/feed/shrinkage-profiles`)

Таблица профилей с колонками: *Что сохнет* (ингредиент или тип корма), *Склад*, *Процент/период*, *Предел*, *Активен*. Действия: редактировать, деактивировать.

### 8.3 Отчёт «Потери от усушки» (`/reports/feed-shrinkage`)

Фильтры: период, склад, ингредиент/тип корма. Таблица:

| Позиция | Начальная масса | Списано по усушке | % | Стоимость потерь |
|---|---|---|---|---|

### 8.4 Баннер в формах расхода

В формах создания `feed_raw_consumption` и `feed_product_shipment` — мягкое предупреждение, если планируемое количество превышает «остаток с учётом прогнозируемой усушки на дату отгрузки»: *«На 2026-05-01 остаток составит ≈ 847 кг (после учёта усушки 12 кг). Вы планируете списать 900 кг».*

## 9. Зависимости и риски

**Риск: расхождение с инвентаризацией.** Раз в квартал фактический остаток на складе сверяется с учётным. Если расхождение > 1% (помимо уже списанной усушки), надо дать инструмент «скорректировать профиль» — например, процент был занижен. План: в отчёте по инвентаризации показывать «списано по усушке за период» и «расхождение по факту». Если руководитель видит, что расхождение систематически больше списанной усушки, он повышает процент в профиле.

**Риск: партия получила усушку, а потом её полностью вернули поставщику.** Возврат = расход (`feed_raw_consumption` с типом «возврат») → остаток уходит в 0 → state замораживается автоматически. Начисленная усушка остаётся в журнале как реальная потеря.

**Риск: изменение профиля задним числом.** Редактирование `percent_per_period` задним числом ретроактивно **не применяется** — старые списания не переигрываются. Если нужно пересчитать с нуля — отдельная админская процедура `POST /feed/shrinkage/reset-lot/{state_id}`, которая откатывает все `stock_movements` этой партии с `kind='shrinkage'` и запускает алгоритм заново с `last_applied_on = arrival_date`. Требует пермишн `feed_shrinkage_run.execute` + подтверждение.

**Риск: производительность.** При ~10 организациях × 50 ингредиентов × 20 активных партий × 1 цикл/сутки получаем ~10 000 расчётов — дёшево. Индексы: `(organization_id, lot_type, is_frozen)` на state, `(organization_id, target_type, ingredient_id, feed_type_id, warehouse_id)` на profiles.

**Зависимость: `warehouses` должны существовать.** Если в организации ещё не настроены склады — профили работают без привязки к складу.

## 10. План внедрения

1. **Миграция** (backend, 1 день): две таблицы, CHECK-расширение, permissions.
2. **Модели + схемы + репозитории** (backend, 0.5 дня).
3. **Сервис `FeedShrinkageService`** с алгоритмом 5.1 (backend, 1–1.5 дня).
4. **Taskiq cron-джоба** `apply_feed_shrinkage_task` (backend, 0.5 дня).
5. **API endpoints** (backend, 0.5 дня).
6. **Unit-тесты:** компаунд vs линейное, замораживание по max %, замораживание по дням, идемпотентность ручного запуска, корректная привязка профиля (склад-специфичный приоритетнее общего) (backend, 1 день).
7. **Frontend:**
   - Страница профилей (0.5 дня).
   - Виджет на партии + график (0.5 дня).
   - Отчёт «Потери от усушки» + экспорт (0.5 дня).
   - Баннер в формах расхода (0.3 дня).
8. **Документация + туториал для бухгалтерии** (0.3 дня).

**Итого:** ≈ 7 рабочих дней одним разработчиком.

## 11. Что осталось решить

- Нужно ли хранить **температуру и влажность склада** (чтобы усушка считалась условно, а не по константе)? Отложим до v2.
- Нужно ли **разные профили для разных сезонов** (зимой суше, летом влажнее)? Можно решить сейчас полем `applies_from_month / applies_to_month`, но усложняет UI. Предложение: v1 — без сезонности, v2 — добавить, если пользователи попросят.
- Должна ли усушка **автоматически попадать в P&L** как расход по счёту «Потери при хранении»? Зависит от плана счетов у клиента. Предложение: v1 — только движение в `stock_movements` + отчёт, без автопроводки; бухгалтер сам принимает решение.
